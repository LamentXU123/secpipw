# secured_pip 中文文档

[English](./README.md) | 简体中文

`secured_pip` 是一个面向 Python 包安装过程的轻量级安全防护工具。它可以作为 `pip` 的包装器使用，在执行包安装时检查潜在的供应链风险，从而降低安装恶意包、拼写仿冒包或可疑依赖的风险。

项目提供的命令为 `spip`。在日常使用中，可以使用：

```bash
spip install requests
```

替代：

```bash
pip install requests
```

这样可以在尽量保持原有 `pip` 使用习惯的同时，为包安装流程增加一层安全检查。

## 项目目标

Python 生态中的供应链攻击一直是重要的安全问题。攻击者可能通过以下方式诱导开发者安装恶意包：

* 发布与热门包名称相似的拼写仿冒包；
* 在依赖中引入可疑的 Direct URL 或 VCS URL；
* 利用刚发布不久、尚未经过充分验证的新版本包；
* 在包元数据中加入可疑链接；
* 通过 `.pth` 文件在安装后触发异常逻辑。

`secured_pip` 的目标不是完全替代所有专业供应链安全工具，而是提供一个更轻量、低学习成本、适合日常开发和 CI 场景的安装防护层。

## 安装方式

可以通过 `pip` 安装：

```bash
pip install secured_pip
```

安装完成后，可以直接使用：

```bash
spip install <package-name>
```

例如：

```bash
spip install requests
```

## 将 pip 映射为 spip

如果希望尽量保持原有使用习惯，可以设置命令别名，让日常输入 `pip` 时实际调用 `spip`。

### Windows Command Prompt

```bat
pip install secured_pip
doskey pip=spip $*
```

### Linux Bash

```bash
pip install secured_pip
echo "alias pip='spip'" >> ~/.bashrc
source ~/.bashrc
```

### macOS Zsh

```bash
pip install secured_pip
echo "alias pip='spip'" >> ~/.zshrc
source ~/.zshrc
```

## 工作方式

执行安装命令时，`secured_pip` 会复用 `pip` 自身的依赖解析流程，先获取本次安装计划，然后在真正构建或安装包之前，对解析出的包进行安全检查。

如果检查通过，安装流程会继续执行。

如果检测到潜在风险，工具会根据风险等级给出警告、询问用户确认，或者要求用户显式忽略警告后才继续安装。

除安装相关命令外，`spip` 的行为会尽量保持与原始 `pip` 一致，因此在大多数场景下可以作为 `pip` 的近似替代工具使用。

## 警告等级

目前项目中的安装警告策略分为三个等级：

| 等级     | 行为                                     |
| ------ | -------------------------------------- |
| HIGH   | 暂停安装，需要用户显式添加 `--ignore-warning` 后才能继续 |
| MEDIUM | 安装前提示用户确认是否继续                          |
| LOW    | 输出警告，但默认继续安装                           |

默认敏感度为 `low`。如果希望使用更严格的策略，可以通过 `--sensitivity` 调整：

```bash
spip install requests --sensitivity medium
```

或：

```bash
spip install requests --sensitivity high
```

不同敏感度下的行为如下：

* `--sensitivity medium`：`MEDIUM` 及以上风险会暂停安装，`LOW` 风险会提示确认；
* `--sensitivity high`：`LOW` 及以上风险都会暂停安装。

## 缓存机制

`secured_pip` 默认会在用户缓存目录中保存部分历史信息，例如：

* PyPI 包名；
* 包发布时间；
* 维护者邮箱历史记录。

这些缓存可以在不同项目之间复用，用于辅助判断包信息是否发生异常变化。

如果需要自定义缓存目录，可以设置环境变量：

```bash
SPIP_CACHE_DIR=/path/to/cache
```

## 当前主要检查项

### 1. 拼写仿冒包检查

攻击者可能发布与热门包名称非常相似的恶意包，例如：

* `requsets` 与 `requests`
* `panda` 与 `pandas`
* `sixth` 与 `six`

`secured_pip` 会在依赖解析后，对低热度包名和本地热门包列表进行比较，从而发现可能的拼写仿冒风险。

### 2. Direct URL 依赖检查

如果安装目标或解析出的依赖使用了 Direct URL、VCS URL 或 PEP 508 direct reference，工具会给出风险警告。

这类依赖可能绕过常规包索引流程，因此在供应链安全场景下需要额外关注。

### 3. 新发布版本检查

如果选中的 PyPI 版本发布时间非常近，工具会提示风险：

* 发布不到 8 小时：`MEDIUM` 警告；
* 发布不到 48 小时：`LOW` 警告。

刚发布的版本可能还没有经过充分使用和审查，因此需要谨慎安装。

### 4. 空描述检查

如果 PyPI 元数据中缺少 summary 和 long description，工具会给出 `LOW` 警告。

缺少基本描述并不一定代表恶意，但可能说明该包质量较低或信息不完整。

### 5. 可疑元数据 URL 检查

如果 PyPI 元数据中包含短链接、裸 IP、可疑顶级域名、嵌入式凭据等 URL，工具会给出警告。

这类链接可能用于隐藏真实跳转目标，或引导用户访问不可信资源。

### 6. 仓库名称不匹配检查

如果 PyPI 元数据指向 GitHub 或 GitLab 仓库，但仓库名称与包名明显不相关，工具会提示潜在风险。

这可以帮助发现包信息伪装、仓库替换或元数据异常等情况。

### 7. 维护者邮箱域名漂移检查

如果某个包的维护者邮箱域名与本地历史缓存相比发生变化，工具会给出 `LOW` 警告。

维护者信息变化并不一定代表攻击，但在供应链安全中属于值得关注的信号。

### 8. 零版本检查

如果选中的包版本为 `0.0` 或 `0.0.0`，工具会给出 `LOW` 警告。

这类版本号可能表示包尚未成熟，也可能出现在异常发布场景中。

### 9. `.pth` 文件检测

部分恶意包可能不会直接在主要代码中植入恶意逻辑，而是通过 `.pth` 文件在安装后触发额外行为。

`secured_pip` 会在安装后的文件系统差异中检查可疑 `.pth` 文件。如果发现风险，会给出 `MEDIUM` 警告，并询问是否删除可疑文件。

## 适用场景

`secured_pip` 适合以下场景：

* 日常 Python 开发环境；
* 个人项目依赖安装；
* CI/CD 中的依赖安装流程；
* 对供应链安全有基本要求，但不希望引入过重工具链的项目；
* 希望以较低成本降低恶意包安装风险的开发者。

## 注意事项

`secured_pip` 可以降低部分供应链攻击风险，但不能保证完全阻止所有恶意包。对于生产环境、关键系统或高安全要求项目，仍建议结合以下措施：

* 固定依赖版本；
* 使用 lock file；
* 审查新增依赖；
* 使用可信包源；
* 在隔离环境中构建和测试；
* 配合静态分析、沙箱分析或专业供应链安全工具。
